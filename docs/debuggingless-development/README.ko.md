# 디버깅리스 개발 (Debuggingless Development)

[🇺🇸 English](README.en.md) | 🇰🇷 한국어

디버깅의 필요성을 최소화하도록 시스템을 설계하는 개발 철학.

---

## 개념

많은 소프트웨어 프로젝트에서 디버깅은 일반적인 개발 과정으로 여겨진다.

일반적인 개발 흐름은 다음과 같다.

코드 작성 → 실행 → 버그 발견 → 디버깅 → 수정 → 반복

Debuggingless Development는 이 흐름을 다른 방식으로 접근한다.

핵심은 **디버깅을 통해 문제를 해결하는 것이 아니라, 애초에 버그가 발생하기 어려운 구조를 설계하는 것**이다.

---

## 핵심 원칙

### Architecture First

시스템 구조는 가능한 한 **잘못된 상태가 표현되지 않도록** 설계해야 한다.

좋은 아키텍처는 특정 종류의 버그 자체를 발생하지 못하게 만든다.

---

### Deterministic Execution

핵심 시스템은 예측 가능한 실행 순서를 가져야 한다.

결정론적 실행은 다음을 쉽게 만든다.

- 시스템 동작 이해
- 문제 재현
- 회귀 버그 탐지

---

### Structural Logging

로그는 임시 디버깅 출력이 아니라 **시스템 구조의 일부**여야 한다.

핵심 이벤트는 항상 로그를 남기도록 설계하여 실행 흐름을 쉽게 추적할 수 있도록 한다.

---

### Debugging as a Last Resort

브레이크포인트나 스텝 디버깅 같은 도구는 여전히 유용하지만  
가능하면 **마지막 수단**으로 사용하는 것이 바람직하다.

대부분의 문제는 다음을 통해 발견된다.

- 아키텍처 제약
- 결정론적 실행
- 구조적 로그

---

## 결론

디버깅을 완전히 제거할 수는 없다.

하지만 강한 아키텍처 설계를 통해 **디버깅이 필요한 상황 자체를 크게 줄일 수 있다**.

많은 경우 가장 좋은 디버깅 전략은  
**버그가 생기기 어려운 구조를 설계하는 것**이다.
